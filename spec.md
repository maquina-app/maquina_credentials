# Spec: `Fragua::Credentials`

**Status:** Draft  
**File:** `lib/fragua/credentials.rb`  
**Test:** `test/fragua/credentials_test.rb`

---

## Goal

A zero-dependency Ruby class that encrypts and decrypts a `credentials.yml.enc` file using AES-256-GCM. Ships inside the container image; the key arrives at runtime via `FRAGUA_MASTER_KEY`. Reading any absent key returns an empty string and never raises.

---

## Background Research: Issues & PRs Consulted

The following problems from the Rails credentials ecosystem informed every design decision below.

### 1. Ruby OpenSSL IV-before-Key nonce-zeroing bug

**Source:** Adam Caudill security post (2016), confirmed in Ruby OpenSSL.

In Ruby's `OpenSSL::Cipher`, if you call `cipher.iv = ...` **before** `cipher.key = ...`, the IV is silently reset to all-zeros after the key is set. The result: every encryption operation uses `\x00 * 12` as the nonce regardless of what you passed in. With AES-GCM, nonce reuse under the same key is catastrophic — an attacker who obtains two ciphertexts can XOR them to recover the XOR of plaintexts, and can also forge authentication tags.

**Fix applied:** Always set `cipher.key` before `cipher.iv`. The current draft already does this. This ordering must be locked down and covered by a test that catches regression.

### 2. AES-GCM nonce reuse is a write-time risk, not a read-time risk

**Source:** RFC 8452 (AES-GCM-SIV), elttam research.

For a credentials file that is written infrequently, the statistical birthday collision risk for a 96-bit random nonce is negligible. However: every call to `Credentials.write` must generate a fresh `SecureRandom`-based IV. It must never re-read the existing IV from the file and reuse it. The current implementation uses `cipher.random_iv` which is correct.

### 3. YAML deserialization can execute arbitrary code

**Source:** CVE-2013-0156, CVE-2022-32224, Trail of Bits (2025), multiple RCE disclosures.

`YAML.load` (and `YAML.unsafe_load`) can instantiate arbitrary Ruby objects. If the `.enc` file is corrupted or tampered with by an attacker who has write access to the file system, loading it with `YAML.load` could execute arbitrary code even though the _encryption_ is intact. The file's HMAC authentication tag only protects confidentiality and integrity of the bytes — it does not constrain what classes are instantiated after decryption.

**Fix required:** Replace `YAML.safe_load(decrypted)` with explicit permitted classes. Credentials files contain only plain scalars, strings, and integers — no Ruby objects need to be deserialized.

```ruby
# Current (safe but could be more explicit):
YAML.safe_load(decrypted)

# Required (defense-in-depth — reject all Ruby objects, allow Symbol for YAML key style):
YAML.safe_load(decrypted, permitted_classes: [], symbolize_names: false)
```

The `permitted_classes: []` call means: reject any `!ruby/object:` tag. This prevents gadget chain execution even if the plaintext was replaced by an attacker.

### 4. Key derivation with raw SHA-256 is not a KDF

**Source:** Rails MessageEncryptor internals; Michael Coyne (2018) on PBKDF2 vs HKDF; Rails KeyGenerator.

The current implementation does:

```ruby
OpenSSL::Digest::SHA256.digest(raw_key)
```

SHA-256 is a hash function, not a key derivation function (KDF). Using it for key derivation from a passphrase has no iteration count, no salt, and no domain separation. An attacker with the hash output learns the full key immediately.

However: `FRAGUA_MASTER_KEY` is intended to be a random 32-byte hex string generated by `SecureRandom.hex(32)` — not a user-entered passphrase. For random-key inputs, HKDF (RFC 5869) is the correct KDF: it performs an "extract-then-expand" step that provides domain separation and allows subkey derivation.

**Fix required:** Replace the SHA-256 shortcut with HKDF when the key is not already exactly 32 bytes:

```ruby
def self.derive_key(raw_key)
  return raw_key.b[0, KEY_LENGTH] if raw_key.bytesize == KEY_LENGTH

  # HKDF-SHA256: extract (salt=nil uses hash-length zero salt per RFC 5869) then expand
  OpenSSL::KDF.hkdf(
    raw_key,
    salt:   "",
    info:   "fragua-credentials-v1",
    length: KEY_LENGTH,
    hash:   "SHA256"
  )
end
```

The `info` label provides domain separation — a key used for Fragua credentials cannot be trivially reused to derive keys for other purposes.

### 5. Encrypted file can be silently cleared on write failure

**Source:** Rails issue #31286 (`Credentials cleared when running credentials:edit`); Rails issue #28143 (permission errors on temp file).

Rails' `EncryptedFile#writing` writes a temp file, then copies it to the destination. If the editor exits before flushing the IO buffer, or if a file permission issue arises mid-write, the destination file can be truncated or overwritten with an empty or partial payload. The credentials are lost.

**Fix required:** Use a write-then-rename pattern (atomic write):

```ruby
def self.write(hash, credentials_path: "config/credentials.yml.enc")
  payload   = YAML.dump(deep_stringify(hash))
  encrypted = encrypt(payload, master_key)

  tmp_path  = "#{credentials_path}.tmp.#{Process.pid}"
  File.write(tmp_path, encrypted)
  File.rename(tmp_path, credentials_path)
ensure
  File.delete(tmp_path) if tmp_path && File.exist?(tmp_path)
end
```

`File.rename` is atomic on POSIX systems when source and destination are on the same filesystem. The temp file is namespaced with `Process.pid` to avoid collisions under parallel workers.

### 6. Integer and mixed-type keys cause silent `nil` reads

**Source:** Rails issue #51215 (`Fix supporting encrypted configuration with integer keys`); Rails PR #42358 (`Credentials: support hash style access in more cases`).

Rails' `deep_transform` broke when credential YAML contained integer keys (e.g. year-keyed data: `2024: abc123`). The transform called `.to_s` on string keys but ignored integer keys, causing lookups to return `nil`.

**Fix required:** `deep_stringify` must handle integer keys explicitly, converting them to strings so all lookups go through the same string path:

```ruby
def self.deep_stringify(obj)
  case obj
  when Hash  then obj.transform_keys(&:to_s).transform_values { |v| deep_stringify(v) }
  when Array then obj.map { |v| deep_stringify(v) }
  else obj
  end
end
```

The current implementation already does this (the `transform_keys(&:to_s)` call handles integers). This behavior must be explicitly tested.

### 7. File permission on the `.enc` file is not set on write

**Source:** General credentials security practice; Rails secrets permission issues.

When `File.write` creates a new file, it uses the process `umask`. In many production environments, this could result in world-readable credentials files (e.g., mode `0644`). The `.enc` file is safe to be world-readable _if_ the encryption key is not compromised, but defense-in-depth calls for `0600` permissions.

**Fix required:** Set file permissions explicitly after write:

```ruby
File.chmod(0o600, credentials_path)
```

### 8. Key missing at read time should fail loudly, not silently

**Source:** Rails `require_master_key` config option (Rails PR #30067).

Rails added `require_master_key` specifically because silent `nil` returns on missing keys led to applications booting in a degraded state without any error, then failing at the point of use with confusing `NoMethodError` or empty-string behavior.

**Decision for Fragua:**

- `read("key")` returns `""` when the _key path_ is absent — this is the intended behavior.
- `MasterKeyMissing` is raised immediately when `FRAGUA_MASTER_KEY` is not set — this should be raised at first use (lazy), not at boot, so tests and dev environments that do not need credentials do not need to set the env var.
- `DecryptionFailed` is raised when the key is present but the file cannot be decrypted — this is always a hard error.

### 9. No mechanism to verify the file is intact before decrypting

**Source:** General defense-in-depth; GCM auth tag behavior.

AES-256-GCM's auth tag already provides integrity verification — if the ciphertext or IV has been tampered with, `OpenSSL::Cipher::CipherError` is raised during decryption. The current implementation already maps this to `DecryptionFailed`. No additional MAC is needed.

However: `Base64.strict_decode64` will raise `ArgumentError` on malformed input (e.g., a truncated file). This must be caught and re-raised as `DecryptionFailed`.

## Requirements

### Reading

- `Credentials.new.read("key")` → decrypted value as a String, or `""` if absent
- `Credentials.new.read("a.b.c")` → dot-path traversal through nested hash
- `Credentials.new.read(nil)` → `""`
- `Credentials.new.read("")` → `""`
- File does not exist → returns `""` on every `read` (does not raise)
- `FRAGUA_MASTER_KEY` absent → `MasterKeyMissing` raised on first `read`
- Correct key, wrong file content → `DecryptionFailed` raised on first `read`
- Decryption result cached per instance (decrypt once per process)

### Writing

- `Credentials.write(hash)` encrypts the hash and writes `credentials.yml.enc`
- Write is atomic: temp file renamed into place, temp file cleaned up on error
- Output file permissions set to `0600`
- Fresh random IV generated on every write (never reuses previous IV)

### Encryption

- Cipher: AES-256-GCM
- IV: 12 bytes, `SecureRandom`-based, generated fresh per write
- Auth tag: 16 bytes
- Wire format: `Base64.strict_encode64(iv[12] + ciphertext + auth_tag[16])`
- Key assignment order: `cipher.key` set before `cipher.iv` (prevents Ruby OpenSSL nonce-zeroing bug)
- Key derivation: HKDF-SHA256 when raw key ≠ 32 bytes; raw bytes used directly when exactly 32 bytes
- HKDF info label: `"fragua-credentials-v1"` (domain separation)

### YAML

- Serialize with `YAML.dump`
- Deserialize with `YAML.safe_load(decrypted, permitted_classes: [], symbolize_names: false)`
- `deep_stringify` converts all keys (including integers) to strings before encrypt
- Credentials are read back as string-keyed hashes; dot-path lookup uses string keys

### Errors

| Error class | When raised |
|---|---|
| `Fragua::Credentials::MasterKeyMissing` | `FRAGUA_MASTER_KEY` env var is unset or empty |
| `Fragua::Credentials::DecryptionFailed` | Wrong key, tampered file, truncated file, or malformed Base64 |

No other exceptions should escape the public API. All `OpenSSL::Cipher::CipherError`, `ArgumentError` (Base64), and `Psych::` errors must be caught and re-raised as `DecryptionFailed`.

## Out of Scope (v1)

- Key rotation / fallback decryptors (Rails' `MessageEncryptor#rotate` pattern) — can be added later
- Environment-specific files (`credentials/production.yml.enc`) — single file only
- CLI editor (`credentials:edit`) — write-time is code-only via `Credentials.write`
- Per-key access control
- Secret scanning / credentials template file generation (Rails issue #37087) — out of scope; the container workflow makes a template unnecessary

## Implementation Notes

### Key ordering guard (critical)

```ruby
cipher.encrypt
cipher.key = derive_key(raw_key)   # key FIRST
cipher.iv  = iv                     # iv SECOND — never swap
cipher.auth_data = ""
```

### HKDF derivation

```ruby
require "openssl"

OpenSSL::KDF.hkdf(
  raw_key,
  salt:   "",
  info:   "fragua-credentials-v1",
  length: 32,
  hash:   "SHA256"
)
```

`OpenSSL::KDF.hkdf` is available in Ruby's stdlib `openssl` gem (OpenSSL 1.1+). No additional dependency needed.

### Atomic write

```ruby
tmp = "#{path}.tmp.#{Process.pid}"
File.write(tmp, encrypted)
File.rename(tmp, path)
File.chmod(0o600, path)
ensure
  File.delete(tmp) if File.exist?(tmp) rescue nil
```

### YAML safe load (explicit empty permit list)

```ruby
YAML.safe_load(decrypted, permitted_classes: [], symbolize_names: false)
```

## Tests Required

| Test | What it asserts |
|---|---|
| `read` with present key | Returns correct string value |
| `read` with absent key | Returns `""` |
| `read` with nil | Returns `""` |
| `read` with dot-path | Traverses nested hash correctly |
| `read` with integer key in YAML | Returns correct value (not nil) |
| Missing file | Returns `""` on every read |
| Missing master key | Raises `MasterKeyMissing` |
| Wrong key | Raises `DecryptionFailed` |
| Truncated/malformed file | Raises `DecryptionFailed` |
| Tampered ciphertext | Raises `DecryptionFailed` |
| Round-trip write→read | Decrypted value matches original |
| IV uniqueness | Two writes produce different IVs |
| Key set before IV | IV extracted from blob is not all-zeros |
| File permissions | Written file has mode `0600` |
| Atomic write on error | Temp file cleaned up; original file untouched on failure |
| Instance caches result | Decrypt called once across multiple `read` calls |

## Container Workflow (unchanged)

```
# 1. Generate master key once — store as an env secret, never in the image
ruby -e "require 'securerandom'; puts SecureRandom.hex(32)"

# 2. Write credentials locally
FRAGUA_MASTER_KEY=<key> ruby -e "
  require_relative 'lib/fragua/credentials'
  Fragua::Credentials.write({
    anthropic_key: 'sk-ant-...',
    database: { password: 'prod-secret' }
  })
"

# 3. Commit credentials.yml.enc into the image — the key never goes in
docker build -t fragua-agent .

# 4. Inject key at runtime
docker run -e FRAGUA_MASTER_KEY=<key> fragua-agent
# or via orchestrator secrets (Fly, K8s Secret, etc.)
```

## Non-Goals

- This class has no opinion on which credentials are required. Callers validate presence.
- It does not know about Rails environments, RAILS_ENV, or any framework-level lifecycle.
- It does not log anything. Callers decide what to surface.
